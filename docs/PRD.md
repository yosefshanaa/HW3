## 1. Overview & Context

### 1.1 Goal
Build a **production-style, agent-based content factory** that turns a single
topic string into a finished, professionally typeset **~15-page mini-book**
(PDF). The system orchestrates a **CrewAI** team of specialised agents
(Researcher → Writer → Reviewer → LaTeX Engineer) and a **LaTeX (LuaLaTeX)**
typesetting layer, demonstrating the lecture's thesis: *"a production agent is a
system, not a prompt."*

### 1.2 User problem
Producing a structured, citation-backed, well-typeset booklet by hand is slow
and error-prone — especially a **bilingual Hebrew/English (BiDi)** document that
must mix right-to-left prose with left-to-right code, formulas and figures.
Authors need a repeatable pipeline that researches a topic, drafts and reviews
the prose, and emits compile-ready LaTeX with all structural elements in place.

### 1.3 Chosen subject of the produced book
**"How to Correctly Build a Start-up"** (כיצד לבנות סטארט-אפ נכון) — a practical
mini-book covering idea validation, the Lean loop, team, product-market fit,
unit economics, fundraising and common failure modes.

### 1.4 Target audience
1. **Course evaluator** — grades the software project (architecture, tests,
   docs, Git history) *and* the produced PDF (structure, BiDi, required
   elements).
2. **End user / author** — runs the CLI to generate a book on a chosen topic.
3. **Future maintainer** — extends the crew, swaps the LLM provider, or retargets
   the LaTeX template.

### 1.5 Market / need note
This mirrors the 2026 agent ecosystem (lecture + architecture deck): CrewAI is
the recommended fit for **role-based agent teams with bounded autonomy and clear
task boundaries**. The project is a concrete instance of the "article/book
writing crew" pattern from the course materials, hardened toward production
(config, gatekeeper, observability hooks, tests).

### Acceptance criteria
- `uv run startup-book build` produces `latex/book.pdf` from a clean checkout
  (given a valid `.env` and an installed LuaLaTeX).
- The PDF contains **all** required elements (§3.1) and compiles its bibliography
  with linked citations.
- `docs/PRD.md`, `docs/PLAN.md`, `docs/TODO.md` and per-mechanism PRDs exist and
  are consistent with the code.

## 3. Requirements

### 3.1 Functional requirements — the produced book (assignment §13.1)
The output PDF **must** contain:

- **FR-B1** Cover page: title, author, date, course, lecturer.
- **FR-B2** Table of contents, chapter structure, **page headers & footers**.
- **FR-B3** ≥ 1 **image** (photo/illustration).
- **FR-B4** ≥ 1 **graph generated by Python code** (matplotlib → vector PDF).
- **FR-B5** ≥ 1 **table**.
- **FR-B6** ≥ 1 **"fancy" mathematical formula** (real math, not plain text).
- **FR-B7** ≥ 1 chapter demonstrating correct **Hebrew↔English BiDi** (RTL/LTR).
- **FR-B8** A **TikZ** diagram (block/flow diagram).
- **FR-B9** A **bibliography** at the end (`.bib` + biber) with **linked
  citations**.

### 3.2 Functional requirements — the system
- **FR-S1** Accept a topic via CLI/SDK and run the crew to produce chapter content.
- **FR-S2** Researcher gathers key facts; Writer drafts chapters; Reviewer
  polishes for accuracy/clarity; LaTeX Engineer maps content into `.tex`.
- **FR-S3** Content flows between tasks via CrewAI `context` (no manual copy).
- **FR-S4** Generate the Python figure(s) programmatically.
- **FR-S5** Compile the LaTeX project to PDF via a scripted multi-pass build.
- **FR-S6** All OpenAI calls pass through the **API Gatekeeper** (rate-limited).
- **FR-S7** All tunables come from **config files**; secrets from `.env`.
- **FR-S8** Report token usage / cost after a run.

### 3.3 Non-functional requirements
- **NFR-1 Quality:** ruff-clean, ≥85% coverage, ≤150 LOC/file, full docstrings.
- **NFR-2 Modularity:** SDK single entry point; provider/template swappable.
- **NFR-3 Reproducibility:** `uv.lock` pinned; versioned config.
- **NFR-4 Security:** no secrets in code or Git; `.env` git-ignored.
- **NFR-5 Observability:** structured logs of crew steps, costs, build passes.
- **NFR-6 Portability:** LaTeX source compiles on TeX Live (Linux/WSL — the
  v1.0.0 build) and MiKTeX (Windows) with the same LuaLaTeX engine.

## 4. User Stories & Use Cases

- **US-1** *As an author*, I run `startup-book build --topic "..."` and get a
  finished PDF, so I don't typeset by hand.
- **US-2** *As an evaluator*, I read `docs/` and the Git history to verify the
  professional process was followed.
- **US-3** *As a maintainer*, I swap `gpt-4o-mini` for another model by editing
  `config/setup.json`, with no code change.
- **US-4** *As a budget owner*, I read the token/cost summary to estimate spend.

**Primary use case (happy path):** `build` → config+secrets loaded → crew runs
(research→write→review→latexify) through the gatekeeper → figures generated →
LaTeX compiled (4 passes) → `latex/book.pdf` + cost report.

## 5. Assumptions, Dependencies, Constraints, Out-of-Scope

**Assumptions**
- A valid `OPENAI_API_KEY` is available in `.env`.
- A LuaLaTeX engine (TeX Live on Linux/WSL, or MiKTeX on Windows) is installed
  and on `PATH`, with a Hebrew OpenType font (e.g. David CLM via `fonts-culmus`).

**Dependencies**
- External: OpenAI API; TeX Live/MiKTeX (LuaLaTeX, biber); Python 3.12 (via `uv`).
- Libraries: `crewai`, `matplotlib`, `python-dotenv`, `pydantic`, `pytest`,
  `pytest-cov`, `ruff`.

**Constraints**
- `sudo` requires the user's password → the LaTeX install is run by the user.
- CrewAI may not support Python 3.14 → pin a 3.12 virtualenv via `uv`.

**Out of scope**
- A GUI (CLI/SDK only). Real-time web research tools (optional; off by default —
  research grounded in curated facts to keep builds deterministic and cheap).
- Languages other than Hebrew/English. Auto-publishing the PDF anywhere.
